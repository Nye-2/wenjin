# Wenjin Research Workbench UI/UX

> Status: Current design source of truth
> Updated: 2026-07-15

## Product posture

Wenjin is a calm academic collaborator for undergraduates and general master's students. The interface should make demanding research work feel understandable without exposing the machinery. Chat carries intent and steering; the Mission Console carries awareness and user confirmation; Prism carries deliberate document editing.

## Workspace composition

The default workspace is chat-first. The right Mission Console is closed until useful. Starting work causes a brief peek; progress, confirmation, evidence, and team detail expand on demand. The layout must preserve a generous conversation width and avoid a permanent technical dashboard.

On mobile the console is a sheet. The composer, model menu, workspace navigation, and console trigger must never overlap. Resizable desktop panes use stable minimums and restore a sensible default.

## Welcome experience

An empty thread shows a non-persisted, type-specific greeting from Wenjin. It gives three or four good starting directions and invites free-form needs. It does not list every feature or look like a launcher. Math modeling emphasizes uploading the problem; SCI and thesis emphasize the research question, materials, and target.

## Conversation

Assistant thinking must have visible but quiet response: a compact active state, meaningful phase text, and progressive content. Do not render raw chain-of-thought, tool JSON, stack traces, or long internal logs. A mission launch appears as a concise receipt with status and a way to open the console.

Users steer naturally with follow-ups. The interface must never ask them to restart a task merely because they opened a side panel. Side discussions and quoted follow-ups should preserve the main mission context.

## Mission Console

The console answers five questions:

1. What is Wenjin doing now?
2. What has changed since I last looked?
3. Who is working on it?
4. What evidence or outputs exist?
5. Is anything waiting for me?

Default content is a compact current-stage timeline, dynamic worker roster, evidence/artifact counts, and attention items. Trace and technical receipts are lazy details. Empty state is short and does not advertise a catalog of tasks.

Subagent names are generated by the main agent. They may have restrained personality, such as “文献猎手 Nora” or “挑刺编辑”, but should communicate role and avoid noisy repetition.

## Review language

Review is framed as control, not danger. Prefer:

- “待确认”;
- “建议补充证据”;
- “可保存草稿”;
- “与当前版本有冲突”.

Avoid infrastructure labels such as “blocked”, “high risk”, schema ids, or provider exceptions in the default UI. Explain the consequence and available action. Protected claims, citations, and documents require deliberate item-level confirmation; accepted subsets can be saved without forcing an all-or-nothing decision. Independent auditing is never automatic UI ceremony: the user starts it from Chat when a result looks questionable.

## Model control

Use one compact popover for model and reasoning effort. Current models are GPT-5.6 Sol/Terra/Luna, with Terra as default. Effort choices are `low`, `medium`, `high`, and `xhigh`, localized in display if desired. Remove the speed submenu. Show an active indicator during generation without shifting the composer layout.

## Visual system

New surfaces use `--wjn-*` tokens only. The visual direction is institutional but contemporary: neutral surfaces, strong typography, restrained teal/green and warm accent use, crisp borders, and sparse elevation. The theme switch may offer two coherent palettes, but neither should read as a monochrome navy-and-white shell.

- Cards are reserved for repeated items, previews, and modals; page sections remain unframed.
- Radius stays at 8px or below unless an existing control requires otherwise.
- Icons use the installed icon library and always have accessible labels/tooltips.
- Motion communicates state changes and respects reduced motion.
- Color never carries status alone.
- Text must fit Chinese and English labels at desktop and mobile widths.

## Accessibility and resilience

All controls are keyboard reachable with visible focus. Splitters expose ARIA value/help. Dialogs and sheets trap focus and return it on close. Review checkboxes and actions have unambiguous labels. SSE disconnects do not leave fake progress; the client refetches and presents a calm reconnect state.

## Anti-patterns

- workflow/card grids as the primary task entry;
- a permanently open operational sidebar;
- fixed expert teams presented as workflow steps;
- raw agent/tool logs in the default view;
- oversized marketing typography inside work surfaces;
- nested cards, decorative orbs, or ornamental gradients;
- local optimistic state presented as committed research truth;
- forcing a new intake when chat already contains the needed context.

## Acceptance checks

1. Empty workspace welcome is useful and non-persisted.
2. Multi-turn chat starts and steers one Mission without losing context.
3. The console peeks, closes, expands, and restores after refresh.
4. Team, progress, evidence, review, partial save, and history match MissionView.
5. Model/effort menu has no speed option.
6. Desktop and mobile have no overlap or clipped labels.
7. Provider/tool failures are translated into actionable user language.
